# Agent flow map — current architecture

This document is the detailed source of truth for the current system.

## 1. End-to-end flow

Current high-level sequence:

1. `preaudit:live`
2. `preaudit:report`
3. autogenerated `*-audit-intake.draft.json`
4. `audit:live`
5. `orchestrator`
6. specialized agents (`collections`, `sales`, `operations`)

The current codebase applies the live workflow only to:

- `preaudit-agent`
- `audit-agent`

`orchestrator`, `collections`, `sales`, and `operations` keep their existing artifact and demo execution behavior.

---

## 2. Runtime split

Hybrid LLM architecture:

- `preaudit-agent` -> `pi-ai`
- `audit-agent` -> `pi-ai`
- `orchestrator-agent` -> Claude Agent SDK
- `collections-agent` -> Claude Agent SDK
- `sales-agent` -> Claude Agent SDK
- `operations-agent` -> Claude Agent SDK

This split is intentional. Diagnostic stages stay provider-agnostic and comparatively cheap, while orchestration and specialist execution still use the Claude SDK.

## 3. Live preaudit flow

### 3.1 Entry

- Script: [`scripts/live/run-preaudit-live.ts`](../scripts/live/run-preaudit-live.ts)
- Input: `--url=https://...`

### 3.2 Tool harness

Live preaudit uses a deterministic runner-controlled business tool harness in [`src/tools/harness.ts`](../src/tools/harness.ts).

Current tools:

- [`fetch_web_page`](../src/tools/fetchWebPage.ts)
- [`extract_social_profiles`](../src/tools/extractSocialProfiles.ts)
- [`detect_tracking_markers`](../src/tools/detectTrackingMarkers.ts)

Rules:

- tools are runner-invoked, not model-selected
- no bash
- no generic coding-agent harness
- no browser automation yet
- no autonomous tool loop

### 3.3 Synthesized input

The live runner writes a one-record JSON array under:

- `data/clients/preaudit-live.json`

That record includes:

- company name
- website
- inferred industry used by the live runner
- extracted website context
- detected social profiles
- tracking markers

### 3.4 Preaudit execution

The agent:

- validates the input
- derives `client_slug`
- derives `display_run_id`
- builds the prompt
- runs the model through `pi-ai`
- validates output
- applies deterministic SEO / Speed / UX scoring
- records scope/prudence metadata

Relevant files:

- [`src/agents/preaudit-agent.ts`](../src/agents/preaudit-agent.ts)
- [`src/preaudit/contract.ts`](../src/preaudit/contract.ts)
- [`src/preaudit/scoring.ts`](../src/preaudit/scoring.ts)
- [`src/preaudit/scope.ts`](../src/preaudit/scope.ts)
- [`src/preaudit/validateOutput.ts`](../src/preaudit/validateOutput.ts)

### 3.5 Post-success outputs

After successful live preaudit:

- `report.md` can be generated by [`scripts/live/run-preaudit-report.ts`](../scripts/live/run-preaudit-report.ts)
- an audit intake draft is generated automatically through [`src/audit/ingestion/createAuditIntakeDraft.ts`](../src/audit/ingestion/createAuditIntakeDraft.ts)

Draft path pattern:

- `data/clients/<client-slug>-audit-intake.draft.json`

This draft is intentionally incomplete and designed for consultant/client editing before `audit:live`.

## 4. Audit live flow

### 4.1 Entry

- Script: [`scripts/live/run-audit-live.ts`](../scripts/live/run-audit-live.ts)
- Required input: `--intake=...`
- Optional inputs:
  - `--preaudit-run-json=...`
  - `--preaudit-summary=...`

### 4.2 Stage-1 audit ingestion

Audit now has a modular ingestion layer under [`src/audit/ingestion/`](../src/audit/ingestion/):

- [`types.ts`](../src/audit/ingestion/types.ts)
- [`fromPreaudit.ts`](../src/audit/ingestion/fromPreaudit.ts)
- [`fromIntake.ts`](../src/audit/ingestion/fromIntake.ts)
- [`buildAuditInput.ts`](../src/audit/ingestion/buildAuditInput.ts)
- [`createAuditIntakeDraft.ts`](../src/audit/ingestion/createAuditIntakeDraft.ts)

Stage-1 normalized audit input can combine:

- manual business intake
- preaudit validated output
- detected social profiles
- tracking markers

This is designed to extend later with additional modules such as:

- `fromCsv.ts`
- `fromPdf.ts`
- `fromCrm.ts`
- `fromAds.ts`

Those future modules are not implemented yet.

### 4.3 Structured input handoff

The live runner writes normalized audit input to:

- `data/clients/audit-live.json`

Then it sets `AUDIT_INPUT_PATH` so `audit-agent` consumes structured normalized input instead of the demo fixture.

### 4.4 Audit execution modes

`audit-agent` currently supports two modes:

1. Demo mode
   - reads `data/mock/audit.json`
2. Structured-input mode
   - reads `AUDIT_INPUT_PATH`

The demo path remains intact.

Relevant files:

- [`src/agents/audit-agent.ts`](../src/agents/audit-agent.ts)
- [`src/audit/contract.ts`](../src/audit/contract.ts)
- [`src/audit/validateOutput.ts`](../src/audit/validateOutput.ts)

## 5. Artifact layout

### 5.1 New layout for preaudit and audit

New runs for preaudit and audit are stored under:

`artifacts/clients/<client-slug>/<agent>/<display-run-id>/`

Examples:

- `artifacts/clients/morales-propiedades/preaudit/preaudit-morales-propiedades-0001/`
- `artifacts/clients/morales-propiedades/audit/audit-morales-propiedades-0001/`

Internal `run_id` remains unchanged and is still stored in `run.json`.

### 5.2 Latest pointer

Each client/agent folder stores:

- `latest.json`

Example:

```json
{
  "display_run_id": "preaudit-morales-propiedades-0001",
  "run_id": "preaudit-20260413T221056Z-5837759d",
  "path": "artifacts/clients/morales-propiedades/preaudit/preaudit-morales-propiedades-0001"
}
```

### 5.3 Legacy layout

Historical folders under:

- `artifacts/runs/`

remain legacy and untouched. The refactor applies only to new preaudit and audit runs.

### 5.4 Current scope

This client-grouped layout currently applies to:

- preaudit
- audit

It does not yet apply to:

- orchestrator
- collections
- sales
- operations

## 6. Preaudit -> audit linkage

When `audit:live` is run with preaudit context, audit `run.json` can include:

- `source_preaudit_run_id`
- `source_preaudit_display_run_id`
- `source_preaudit_path`

This makes the provenance of structured audit context explicit.

## 7. Inputs and editable files

### Demo fixtures

- `data/mock/` contains demo JSON fixtures

Key demo inputs:

- `data/mock/preaudit.json`
- `data/mock/audit.json`

### Live / editable client inputs

- `data/clients/` contains generated and editable client files

Examples:

- `preaudit-live.json`
- `audit-live.json`
- `*-audit-intake.draft.json`
- manually completed audit intake files

The audit intake draft is autogenerated from a successful live preaudit and is meant to be edited before `audit:live`.

## 8. Scope and prudence layer

Preaudit includes a light scope/prudence classifier:

- `site_type`
- `framework_fit`
- `scope_confidence`

Purpose:

- identify when the SME/service/lead-gen framework is a good fit
- soften certainty when the site looks enterprise, global, platform-like, or otherwise off-scope

The preaudit system is optimized for SME, service-business, and lead-generation websites. It does not claim live benchmark measurements or full enterprise-grade crawling.

## 10. Detailed execution summary

### Preaudit

1. `run-preaudit-live.ts` fetches public website facts through the tool harness
2. writes `data/clients/preaudit-live.json`
3. `runPreauditAgent()` reads the selected input record
4. derives `client_slug` and `display_run_id`
5. writes artifacts to `artifacts/clients/<client-slug>/preaudit/<display-run-id>/`
6. updates `latest.json`
7. can generate `report.md`
8. writes `*-audit-intake.draft.json`

### Audit

1. `run-audit-live.ts` reads manual intake
2. optionally reads source preaudit output and tool facts
3. builds normalized stage-1 audit input via `buildAuditInput(...)`
4. writes `data/clients/audit-live.json`
5. `runAuditAgent()` consumes that file through `AUDIT_INPUT_PATH`
6. derives `client_slug` and `display_run_id`
7. writes artifacts to `artifacts/clients/<client-slug>/audit/<display-run-id>/`
8. updates `latest.json`
9. records preaudit provenance when available

### Orchestrator and specialist agents

No major live-flow refactor has been applied there yet. They still follow their existing demo/mock execution model and artifact behavior.
