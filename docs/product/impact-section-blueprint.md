# Impact Section Blueprint

## 1. Purpose

Impact is the future client-facing workspace section that translates EBC's work into business value.

EBC should not only show that audits were completed, agents were activated, or outputs were delivered. Clients need to understand what those actions mean for the business: what value was identified, what value is being unlocked, what has improved, and what still needs attention.

Impact matters because it:

- Shows progress in business terms.
- Connects workstreams to outcomes.
- Reinforces ROI and the value of EBC's work.
- Gives clients confidence that the platform is not just producing outputs.
- Helps move the workspace narrative from diagnosis to measurable change.
- Makes consulting value visible over time.

The purpose of Impact is to make business improvement legible. It should help clients see the relationship between diagnosis, execution, agents, outputs, and outcomes.

## 2. What Impact Is

Impact is a structured view of business and operational improvement signals inside the EBC workspace.

It should answer questions like:

- What value has EBC identified?
- Which opportunities are being worked on?
- What has improved?
- Which workstreams are tied to which outcomes?
- Where is there enough evidence to claim progress?
- What still needs attention before value can be unlocked?

Impact is not:

- A raw analytics dashboard.
- A clone of Google Analytics, Looker, or a BI tool.
- A noisy metrics page.
- A technical operations surface.
- A vanity metrics display.
- A place to show numbers without business interpretation.

Impact should combine qualitative and quantitative evidence. It should be comfortable saying "value identified" or "value being unlocked" before full measurement exists, as long as the difference between projected and observed value is clear.

## 3. What Should Appear In Impact

Impact should include signals that connect EBC work to client-understandable value.

### Growth Leaks Identified

This belongs because diagnosis often reveals where revenue, conversion, speed, or operational quality is being lost.

Examples:

- Website conversion friction.
- Missed follow-up opportunities.
- Weak handoff between marketing and sales.
- Poor account prioritization.
- Reporting gaps that hide performance issues.

These items should explain the business meaning, not just the finding.

### Workflow Friction Reduced

This belongs because many EBC workstreams improve process quality, speed, consistency, or coordination.

Examples:

- Fewer manual handoffs.
- Clearer approval paths.
- Better CRM hygiene.
- Reduced duplicate work.
- Improved operating visibility.

This category should connect process improvements to business outcomes such as faster response, better execution, reduced errors, or clearer ownership.

### Response Time Improvement

This belongs when EBC helps a client respond faster to leads, customers, internal requests, market changes, or operational issues.

Response time can be measured directly where integrations exist, or described as a projected improvement when measurement is not yet connected.

The key is to explain why speed matters for the client's business.

### Outputs Delivered

This belongs because outputs are tangible evidence of progress, but Impact should not simply count documents.

Outputs should appear when they support a business result:

- A diagnostic report that identifies priority opportunities.
- A playbook that standardizes execution.
- A recommendation memo that enables a decision.
- A CRM cleanup plan that improves reporting reliability.
- Website recommendations that support conversion improvement.

Impact should summarize what the output made possible.

### Workstreams Completed

This belongs because completed workstreams are the clearest unit of delivered transformation.

Impact should show:

- What the workstream targeted.
- What changed.
- What output or decision was produced.
- What business outcome it supports.
- Whether observed results are available yet.

Completion alone is not impact. Completion becomes impact when it is connected to the outcome it was meant to create.

### Agent-Supported Progress

This belongs because agents are part of how EBC scales execution.

Impact should show where agents helped create value, such as:

- Monitoring priority accounts.
- Producing research.
- Summarizing operational risks.
- Drafting recommended actions.
- Identifying missing context.
- Supporting recurring analysis.

The framing should be business-centered: what the agent helped accomplish, not how many runs occurred.

### Visibility Gains

This belongs because sometimes the first value EBC creates is making unclear business conditions visible.

Examples:

- A client can now see where pipeline handoff breaks down.
- A team can now see which website pages need attention.
- Leadership can now see which workstreams are active and blocked.
- Operators can now see which inputs are missing before automation can proceed.

Visibility gains are valid impact signals when they improve decision quality or execution clarity.

### Projected Value

This belongs when EBC has identified likely value but observed results are not yet available.

Projected value should be labeled clearly. It may include estimated time saved, opportunity size, risk reduction, conversion potential, or operational leverage.

Projected value should be conservative and tied to workstreams or recommendations. It should not overstate precision.

### Observed Value

This belongs when there is evidence of measurable change.

Observed value may include:

- Faster response times.
- Improved completion rates.
- Reduced backlog.
- Higher conversion.
- Better data quality.
- Increased output throughput.
- Reduced manual effort.

Observed value should be distinguished from projected value and should link back to the source workstream, output, integration, or evidence.

## 4. What Should Not Appear

Impact should intentionally exclude information that is noisy, technical, or disconnected from business meaning.

The following should stay out:

- Low-level telemetry.
- Raw logs.
- Debugging counters.
- Queue, retry, or infrastructure metrics.
- Token usage or model performance details.
- Vanity metrics with no business meaning.
- Metrics that are not connected to workstreams, outputs, or client goals.
- Internal-only operational alerts.
- Raw event streams.
- Charts included only because data exists.

If a metric cannot answer "why does this matter to the client?", it does not belong in Impact yet.

## 5. Presentation Model

Impact should feel like a business value surface, not a dashboard full of charts.

Possible presentation directions:

- Grouped business themes: organize value by growth, operations, visibility, speed, and risk.
- Before/after modules: show what changed from diagnosis to current state.
- Value summary cards: summarize identified, in-progress, and observed value.
- Qualitative and quantitative mix: combine narrative evidence with measured signals.
- Confidence or maturity indicators: show how strong the evidence is.
- Opportunity vs progress framing: separate value identified from value actively being unlocked.

Recommended direction: an opportunity-to-progress model grouped by business theme.

The first Impact experience should show:

- Value identified: opportunities found through diagnosis, agents, or workstreams.
- Value being unlocked: active workstreams and agent-supported progress tied to outcomes.
- Value observed: measured or confirmed improvements where evidence exists.
- Needs attention: blockers, missing inputs, or unactivated workstreams that prevent value from being realized.

This model fits EBC because it supports both early-stage clients and more mature clients. It does not require every outcome to be measurable on day one, but it still creates a clear path from diagnosis to progress to evidence.

Impact items should use plain business language, concise summaries, and links back to the relevant workstream, output, report, or activity event. Numbers can be included when useful, but the interpretation should come first.

## 6. Relationship To Other Sections

Impact should connect the workspace's work to business value without replacing other sections.

### Dashboard

Dashboard shows the current operating state: what matters now, what needs attention, and recent progress.

Impact shows the value narrative over time. Dashboard may include a small Impact preview, such as top value identified, active value being unlocked, or a recent observed improvement.

### Diagnosis

Diagnosis identifies problems, opportunities, and priorities.

Impact translates those findings into value themes, projected outcomes, and progress evidence. Diagnosis answers "what did we find?" Impact answers "what business value does this point to?"

### Workstreams

Workstreams are the execution units that unlock impact.

Impact should be tied directly to workstreams. Each meaningful Impact item should be able to link to the workstream responsible for unlocking or measuring that value.

### Agents

Agents support research, monitoring, execution, analysis, and recommendations.

Impact should show agent-supported progress only when an agent contributes to a business outcome, output, decision, or workstream. It should not show raw agent activity.

### Activity

Activity shows meaningful workspace events over time.

Impact should use some of the same event foundation, but with a different purpose. Activity answers "what changed?" Impact answers "what value did that change create or unlock?"

### Future Reports

Reports should provide durable, shareable outputs.

Impact should link to reports when those reports support a value claim, document an improvement, or explain an opportunity. Reports can provide evidence; Impact provides the client-facing value interpretation.

## 7. Empty States

Impact should have intentional empty states that preserve trust. It should not invent value before EBC has enough signal.

### No Impact Signals Yet

Use when the client is new and no diagnostic or execution signal exists.

Example:

`No impact signals yet`

`Impact will appear here once EBC has identified meaningful opportunities or progress.`

### Still In Diagnosis

Use when the client is in preaudit, Business Context, or audit and value has not been structured yet.

Example:

`Impact is still being identified`

`EBC is reviewing the business context and diagnosis before summarizing value opportunities.`

### Implementation Not Started Yet

Use when value opportunities have been identified, but no workstreams have started.

Example:

`Value identified, implementation not started`

`Activate a recommended workstream to begin turning these opportunities into progress.`

### Not Enough Evidence Yet

Use when work is active but measured outcomes are not available.

Example:

`Progress is underway`

`EBC needs more evidence before showing measured change. Active workstreams and outputs will appear here as impact signals mature.`

### No Measured Change Yet

Use when projected value exists but observed value has not been confirmed.

Example:

`No measured change yet`

`Projected value is visible, but observed results will appear once the relevant workstreams have enough evidence.`

These states should make the product feel honest. Impact is more credible when it shows the difference between identified, projected, and observed value.

## 8. Future Evolution

Impact can start as a lightweight value narrative and evolve into a richer outcome tracking surface.

Potential future directions:

- Richer analytics: integration-backed measurement for selected business outcomes.
- ROI views: estimated and observed value connected to workstreams, outputs, and client goals.
- Workstream scorecards: progress, evidence, blockers, outputs, and value for each workstream.
- Measurable change over time: trend views for response time, conversion, throughput, quality, or other domain-specific outcomes.
- Client outcome tracking: goals, baselines, milestones, and confirmed improvements.
- Internal vs client impact views: internal teams may see confidence, evidence quality, assumptions, and measurement gaps; clients see curated value language.
- Value confidence levels: distinguish directional, projected, validated, and measured impact.
- Impact summaries: monthly or quarterly value narratives that can be shared with stakeholders.
- Integration-backed evidence: CRM, website analytics, support systems, finance tools, or operational systems feeding observed results.

The long-term opportunity is for Impact to become the value memory of the EBC workspace: the place where identified opportunities, execution progress, and business outcomes come together.

## 9. Recommended First Implementation Slice

Impact v1 should be lightweight, believable, and closely tied to existing EBC product objects.

Recommended first slice:

- Show value themes derived from diagnosis, workstreams, outputs, and agent-supported recommendations.
- Use the opportunity-to-progress model:
  - Value identified.
  - Value being unlocked.
  - Value observed.
  - Needs attention.
- Link each Impact item to a related workstream, output, diagnosis finding, report, or activity event where possible.
- Include qualitative summaries first, with quantitative values only when evidence exists.
- Label projected value separately from observed value.
- Include clear empty states for diagnosis, implementation, and insufficient evidence.
- Avoid raw metrics dashboards, vanity charts, telemetry, logs, and unsupported ROI claims.

The first version should help clients understand why EBC's work matters without pretending that every improvement is already measured. It should make value visible, connect it to workstreams, and create a credible path from diagnosis to execution to outcomes.
