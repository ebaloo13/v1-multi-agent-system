# Project Context

This repository contains B2B business agents built in TypeScript using a hybrid LLM runtime: `pi-ai` for diagnostic agents and the Claude Agent SDK for orchestration and specialist agents.

## Goal
Build realistic, production-oriented mock agents for SME consulting use cases that can later be deployed in real client environments.

## Current Focus
Documented multi-agent consulting workflow:
- Preaudit agent
- Audit agent
- Orchestrator agent
- Collections agent
- Sales agent
- Operations agent

---

## System Philosophy
- Agents are **assistive, not autonomous decision-makers**
- Business logic should be **deterministic where possible (policies, scoring, artifact routing, ingestion normalization)**
- LLMs are used for:
  - summarization
  - reasoning explanation
  - drafting communication
- Every run must be **traceable, reproducible, and auditable**

---

## Current Architecture
The current documented flow is:

1. **Preaudit live**: fast digital diagnostic for SEO, PageSpeed, UX, and tracking on one real website URL
2. **Preaudit report**: deterministic Markdown report generated from validated preaudit output
3. **Business Context draft**: autogenerated editable JSON draft for the next stage
4. **Audit live**: structured business diagnostic using normalized Business Context + optional preaudit context
5. **Orchestrator**: routing and execution layer that runs the audit first, validates routing, and executes the selected specialist agents
6. **Specialized agents**:
   - Collections
   - Sales
   - Operations

Most agents are still mock-data driven in the current phase. Preaudit and audit now have live-oriented input paths, but the system remains focused on safe, deterministic workflows rather than production integrations.

Naming convention:

- Client-facing label: **Business Context**
- Spanish reference: **Contexto del negocio**
- Internal legacy term: `intake`, still used by existing file names, script flags, metadata fields, and ingestion module names

### Execution Layout
- `scripts/demo/` contains mock/demo single-run entrypoints
- `scripts/batch/` contains repeated mock/demo batch entrypoints
- `scripts/live/` contains live preaudit and audit entrypoints plus deterministic report generation
- `data/mock/` contains the current mock JSON fixtures
- `data/clients/` contains generated live client inputs, Business Context drafts stored with legacy `audit-intake` file names, and structured audit inputs
- `src/common/` contains lightweight shared helpers such as run naming and client artifact path helpers
- `src/tools/` contains the deterministic preaudit fact-collection harness and business-oriented tools
- `src/audit/ingestion/` contains modular stage-1 audit ingestion builders

### Artifact Layout
- New preaudit and audit runs are stored under `artifacts/clients/<client-slug>/<agent>/<display-run-id>/`
- `latest.json` is maintained per client and agent
- Internal `run_id` remains the traceability key inside `run.json`
- Historical `artifacts/runs/` remains legacy and untouched

### LLM Usage
- `preaudit-agent` and `audit-agent` use `pi-ai` (multi-provider abstraction)
- `orchestrator-agent`, `collections-agent`, `sales-agent`, and `operations-agent` use the Claude SDK

This separation allows controlled migration and evaluation of different model providers without affecting core system behavior.

### Tooling Boundary
- The tool harness is currently for `preaudit` live ingestion only
- Tools are runner-invoked, not model-selected
- Allowed tool scope is business fact collection only:
  - fetch a public webpage
  - extract likely social profile links
  - detect heuristic tracking markers
- Do not turn this into a generic coding agent:
  - no bash tool
  - no edit/write file tool exposure
  - no arbitrary shell execution
  - no browser automation yet

---

## Output Requirements
- Must return **valid JSON only** (no markdown fences)
- Must conform to schema
- Must match the agent-specific validated output contract
- Must be concise, auditable, and safe to store in `run.json`
- For preaudit, model-generated scores are not trusted as final; deterministic scoring in code is the source of truth written to `validated_output`

---

## Data Assumptions (Current Phase)
- Demo input data comes from static JSON files under `data/mock/`
- Live preaudit synthesizes a one-record JSON input under `data/clients/` from deterministic tool-harness facts collected from a fetched homepage
- Live preaudit also generates:
  - `report.md`
  - `*-audit-intake.draft.json`
- Live audit can consume structured input built from Business Context + preaudit facts
- No external integrations
- No real browser automation, crawling, or live website measurement yet
- Preaudit scope-fit classification is heuristic and is used only to add prudence, not to block runs
- Preaudit may surface scope warnings for enterprise, global, or otherwise off-scope sites

---

## Engineering Constraints
- TypeScript only
- Deterministic validation using Zod
- Strict parsing (fail on invalid JSON)
- Keep demo/mock execution clearly separated from live execution
- Presentation/reporting layers must remain deterministic and derive only from validated output
- Observability metadata in artifacts may be extended, but existing artifact readers must keep working
- Preaudit and audit live runs may add optional artifact metadata such as:
  - `tools_used`
  - `facts_collection_source`
  - `audit_input_source`
  - `intake_path` (legacy metadata field for the Business Context file path)
  - source preaudit linkage fields
- All runs must generate artifacts:
  - `run.json`
  - `events.ndjson`

---

## Near-Term Roadmap
1. Keep preaudit, audit, orchestrator, and specialist agents aligned with the documented architecture
2. Extend deterministic reporting and ingestion patterns where useful without changing schemas or artifact contracts
3. Extend validation and tests across agents that still lag collections
4. Add real integrations only when requirements are explicit

---

## Anti-Patterns to Avoid
- Over-engineering early
- Embedding business logic only in prompts
- Relying on LLM outputs without validation
- Mixing responsibilities across agents
- Turning presentation/report scripts into business-logic layers
- Letting the model decide arbitrary tool calls before the harness and contracts are ready
- Expanding preaudit tools into repo-editing or shell-execution capabilities
- Treating enterprise or platform sites as if the SME lead-gen framework applies cleanly without qualification

---

## Definition of Done
- Demo commands run from `scripts/demo/` and batch commands run from `scripts/batch/`
- Live preaudit and audit flows run from `scripts/live/`
- Live preaudit produces:
  - validated output
  - client-grouped artifacts
  - `report.md`
  - a Business Context draft, currently stored with the legacy `audit-intake` file name pattern
- Live audit can consume structured input
- Runs produce valid structured outputs
- Saves artifacts per run
- Handles failures safely
- Preserves internal traceability via `run_id` while exposing human-friendly artifact paths via `display_run_id` and client-grouped folders
- Keeps live fact collection deterministic, auditable, and business-focused
- Preserves clear separation between:
  - preaudit as fast digital diagnostic
  - audit as deeper business diagnostic
  - orchestrator as routing / execution
  - collections / sales / operations as specialized execution agents
